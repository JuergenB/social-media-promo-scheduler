Image Size Handling Report

Date: 2026-03-29  |  Related: GitHub Issue #62

TL;DR

Airtable does NOT compress images. A 5-6MB image is stored byte-for-byte at full resolution. The assumption that Airtable auto-reduces file size is incorrect. However, the code currently reads back the thumbnails.large URL (a resized version), not the original — so in practice, published images are smaller than what was uploaded, but this is accidental and causes quality loss.
Zernio DOES compress at publish time. Zernio applies per-platform compression server-side (Sharp for images, FFmpeg for video) before uploading to social platforms. So even oversized images won't fail at the social platform — they'll just be compressed by Zernio.
The real problem: no size validation exists anywhere in the upload path. A 5-6MB file upload will silently fail due to the Next.js 4MB body limit, or succeed but get degraded to a thumbnail. The user gets no meaningful error message.

1. Does Airtable Compress Images?

No. Airtable stores originals at full resolution.

AspectBehavior
Original fileStored exactly as uploaded, no compression, no resizing
ThumbnailsAirtable generates small and large thumbnail renderings for UI display — these are separate files, not modifications of the original
File size limit5GB per file (1GB on Free plan)
Base storage quotaFree: 1GB, Team: 20GB, Business: 100GB, Enterprise: 1TB
Copy-paste caveatCopying between attachment fields can reference thumbnails instead of originals, causing quality loss

What this means for PolyWiz: If someone uploads a 6MB photo to a post's Image Upload field, Airtable stores the full 6MB. But our code on line 51 of /api/posts/[id]/image/route.ts reads back thumbnails.large.url instead of attachment.url — so the Image URL field gets a resized thumbnail, not the original. This is both a hidden quality loss and a hidden size reduction.

3. What Happens Today With a 5-6MB Image?

Scenario A: File upload via campaign detail page

  1. 5MB file is read into memory
  2. Base64 encoding bloats it to ~6.7MB
  3. Next.js 4MB body limit rejects the request (413 or silent truncation)
  4. User sees "Failed to upload image" with no explanation
  5. Result: silent failure

Scenario B: File just under 4MB (e.g., 3MB original = 4MB base64)

  1. Passes the Next.js body limit
  2. Airtable stores the full 3MB original
  3. Code reads back thumbnails.large.url — a resized version
  4. Publish sends thumbnail URL to Zernio
  5. Result: works, but quality degraded by thumbnail extraction

Scenario C: URL pasted (not uploaded)

  1. No upload occurs — just a URL string stored
  2. At publish time, Zernio fetches the URL directly
  3. Zernio applies per-platform compression if needed
  4. Result: works correctly at any image size

Compression uses Sharp (images) and FFmpeg (video) server-side. Original files are preserved; compressed versions are created for the social platform upload only.

5. Zernio Presign Limits

The POST /v1/media/presign endpoint supports files up to 5GB. Since this uploads directly to cloud storage (bypassing the Next.js server), there is no body size limit concern. This is the path used by the compose page but not by the campaign/post image upload flows.

6. Recommendations

Priority 1 — Wire up existing size validation (quick fix) getMaxFileSize() and isValidMediaType() already exist in use-media.ts. Add client-side checks before uploadImageMutation.mutate() in the campaign detail page. Set practical limit at 4MB (not the 10MB the utility defines) to account for base64 bloat. Show a clear error: "Image must be under 4MB."
Priority 2 — Fix thumbnail quality loss In /api/posts/[id]/image/route.ts line 51, change attachment?.thumbnails?.large?.url || attachment?.url to attachment?.url || attachment?.thumbnails?.large?.url. The current order prefers the thumbnail over the original, causing unnecessary quality degradation.
Priority 3 — Switch to Zernio presign for all image uploads (medium-term, aligns with Issue #62) Replace the Airtable base64 upload path with Zernio's presign flow. This eliminates the 4MB body limit, produces permanent URLs (no Airtable expiration), and lets Zernio handle per-platform compression. The infrastructure already exists in use-media.ts.
