Image Storage Options Comparison

Date: 2026-03-29 (updated 2026-03-30)  |  Related: Epic #63 — Image Handling Pipeline

TL;DR

Decision: Vercel Blob

Vercel Blob is included in the existing Vercel Pro plan (already paying $20/mo for deployment). Incremental cost: $0. Simple put()/del() API, permanent public URLs, edge CDN delivery.

Why not Supabase Storage? While Supabase Pro includes 100 GB storage, each additional project incurs $10/month for a dedicated Postgres instance — even if you only use Storage. Since PolyWiz doesn't need Supabase's database, auth, or edge functions, this is pure overhead. A Supabase project (polywiz) was created and tested before this was discovered.

Image profile matters: Most images are scraped web-optimized images (~sub-200KB), not large user uploads. At these sizes, Vercel Blob's Pro plan limits (100 GB storage, unlimited bandwidth) are more than sufficient. The earlier 1MB-per-image estimates over-weighted upload/compression scenarios.

What Proper Image Storage Enables

Capabilities that URL references alone cannot support:

  1. Image permanence — Source sites go down, URLs change. Downloaded and re-hosted images make campaigns resilient.
  2. Platform-specific sizing — Sharp in serverless functions for per-platform transforms: 1080x1350 for Instagram 4:5, 1000x1500 for Pinterest 2:3.
  3. Carousel & PDF generation — Generating LinkedIn carousel PDFs or Instagram carousel sets requires reliable access to source images, not URLs that might 404.
  4. Audit trail — Every image used in every campaign is preserved, even if the source site removes the original.
  5. Email/preview generation — Future email digests or report cards need permanent image URLs you control.
  6. Deduplication — Content-addressed storage can detect duplicate images across campaigns.

Proposed Hybrid Approach

Layer 1: Vercel Blob (primary image store) User-uploaded images are stored in Vercel Blob at upload time. The permanent public URL is stored in Airtable's Image URL field. Scraped web images keep their original URLs (they're already permanent and web-optimized).
Layer 2: Zernio presign (publish-time optimization, optional) At publish time, optionally re-upload to Zernio via presign for maximum reliability. Or just pass the Vercel Blob / original URL directly — both are permanent, so Zernio can fetch them anytime.

Implementation:

  1. Install @vercel/blob — simple put()/del() API
  2. Add lib/blob-storage.ts utility (~30 lines: uploadImage, deleteImage, isBlobUrl)
  3. At manual upload: compress client-side → upload to Vercel Blob → store permanent URL in Airtable
  4. Scraped images: keep original URLs (already web-optimized, sub-200KB typical)
  5. At publish time: pass URL to Zernio (permanent, no refresh needed)
  6. The Airtable Image Upload attachment field becomes unnecessary for new uploads
